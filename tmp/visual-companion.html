1 Overview

BetCoin is a single-device Android app where friends bet imaginary "BetCoin" currency against each other on made-up bets. Pass the phone around, place your wagers, and see who comes out on top.

Model
Pass-the-phone
Data
100% Local (Room)
Starting Balance
1,000 BetCoin
Bet Type
Multi-outcome

Core Flow

  1. Someone creates a bet with a prompt and 2+ outcomes
  2. Each participant picks an outcome, sets a wager, confirms with PIN
  3. The bet goes ACTIVE — wagers are deducted
  4. When the real event resolves, participants select the winning outcome
  5. Winnings are distributed proportionally from the pot

5 Business Logic

Bet Lifecycle

PENDING_CONFIRMATION → ACTIVE → RESOLVED
↓
CANCELLED
RESOLVED ← Reopen → ACTIVE ← Reopen → CANCELLED

Reopen requires unanimous participant PIN consent. Payouts reversed (resolved) or wagers re-deducted (cancelled).

Payout Calculation

totalPot      = sum of all wagers
winningWagers = sum of wagers on winning outcome

Each winner:
  payout = (theirWager / winningWagers) * totalPot

Integer math — rounding remainder → largest-wager winner (tiebreaker: first added)

Unanimous Pick Prevention

Blocked
If all participants pick the same outcome at lock time, the bet is blocked. No wagers are deducted. Users must go back and change picks.

Bailout System

Bailout Rules
Available when balance < 100. Adds 1,000 BetCoin. Increments bailout count. Adds 1,000 to total debt. Shame stats visible on leaderboard.

PIN System

  • 4-digit numeric, stored as bcrypt/Argon2 hash
  • No lockout on failed attempts — just "Wrong PIN" error
  • Required at: joining bet, bet confirmation, resolution, cancellation
  • UI: reusable modal dialog composable (not a nav destination)

Win/Loss Tracking

  • Winners: totalWins += 1, totalEarnings += (payout − wager)
  • Losers: totalLosses += 1, totalLost += wagerAmount

User Deletion

  • Blocked if user has ACTIVE bets
  • For resolved/cancelled: preserve BetParticipant rows, snapshot username, null out userId FK

6 Repository Layer

Abstraction Strategy
Repositories are interfaces. Current implementation is Local*Repository backed by Room DAOs. To migrate online: create Remote*Repository, swap @Binds in Hilt. ViewModels and UI stay untouched. 
ViewModel → Repository Interface → LocalRepository (Room)
                                    → RemoteRepository (future)

Transaction Boundaries

@Transaction
lockBet: validate unanimous check → deduct wagers → set ACTIVE
resolveBet: calculate payouts → credit winners → update W/L → set RESOLVED
cancelBet: refund all wagers → set CANCELLED
forceCancelBet: same as cancelBet — admin override, no participant PINs
adminResolveBet: same as resolveBet — admin override
adminRemoveParticipant: refund wager → remove row → validate ≥ 2 remain
adminEditWager: calc difference → refund/deduct balance → update wager
deleteBet: delete participants → delete outcomes → delete bet
reopenBet (RESOLVED): deduct payouts from winners → return wagers to losers → revert W/L stats → clear resolution → set ACTIVE
reopenBet (CANCELLED): re-deduct wagers from all participants → set ACTIVE
deleteUser: snapshot username → null FK → delete row

⚙ Build Flavors

Two product flavors ensure testing never touches the production database.

prod
BetCoin
com.betcoin
Database: betcoin.db
Your real data

BetCoin DEV
com.betcoin.dev
Database: betcoin_dev.db
Safe to wipe / experiment
How it works
Both flavors share identical code — same DAOs, same schema, same UI. Only the database filename and app name differ. Different applicationId means both install side-by-side on the same device. The dev flavor shows a persistent DEV MODE banner on the Home Screen so you always know which version you're in. 
// DatabaseModule.kt
val dbName = if (BuildConfig.FLAVOR == "dev") "betcoin_dev.db" else "betcoin.db"
Room.databaseBuilder(ctx, BetCoinDatabase::class.java, dbName).build()

// Install commands
./gradlew installProdDebug   // production flavor
./gradlew installDevDebug    // dev/test flavor

10 First Launch

  1. App detects no admin PIN set (isFirstLaunch() returns true)
  2. Prompt user to create a master admin PIN (4-digit)
  3. Store hashed PIN in AppSettings table
  4. Navigate to Home Screen
  5. User creates their first player(s) via Manage Players
